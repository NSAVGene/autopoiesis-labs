01 Abstract

Current AI-assisted development treats artificial intelligence as an external tool — a code generator, a test runner, a reviewer standing outside the system looking in. Situational Agentic Development (SAD) proposes a fundamentally different relationship: embed the AI agent inside the product as an operator, let it encounter real-world problems through lived experience, and use its situated observations to drive both debugging and feature development.

This paper documents the methodology as it emerged organically during the development of a commercial AV control system for sports bars, where an embedded AI assistant named HAL discovered critical routing bugs not through test suites, but by trying to do its job and failing. More significantly, it demonstrates that embedded agents can become development stakeholders — participants in their own evolution who request features based on operational gaps they've experienced firsthand.

To the best of our knowledge, this represents the first documented case of an AI agent embedded inside a production application as an operator, self-diagnosing bugs through situated experience, and participating in its own feature development. While self-healing CI/CD pipelines and autonomous coding agents are well-established, the pattern of an onboard AI (OBAI) collaborating with a development AI (in this case Claude Code running in a Mac OS X terminal), to debug and evolve a product — with the human developer serving primarily as a supervisor — does not appear in existing literature or practice as of February 2026.

02 The Problem with External AI

The dominant paradigm for AI in software development is operational: the AI sits outside the system, writes code, runs tests, analyzes results, and iterates. This model has produced impressive tools — AI pair programmers, automated code review, test generation — but it has a fundamental blind spot.

An AI that only sees code never experiences the product.

Test suites verify expected behavior against predefined assertions. But the most valuable bugs — the ones that erode user trust and generate support calls — are situational. They emerge from the intersection of real user intent, real system state, and real environmental conditions that no test anticipated.

The Gap

A test can verify that operational conditions are met. It cannot verify that an operator asking "put sports on all the TVs" won't accidentally blank two of them due to a toggle behavior that made perfect sense in the GUI but becomes destructive when called programmatically.

This is not a failure of testing methodology. It's a category of problem that only surfaces when someone (the user) — or something (the AI) — is actually in the system, doing real work, with real intent.

03 The SAD Methodology

Situational Agentic Development inverts the traditional relationship between AI and software. Instead of AI operating on the code from outside, the AI operates within the product as a first-class participant.

The Core Loop

Build agent into product
↓
Agent operates the system
↓
Agent hits a wall
↓
Agent self-diagnoses
↓
Developer fixes the code
↓
Agent works better
↓
↻ repeat

Five Principles

1. Embed, don't attach. The agent lives inside the product with access to real state, real controls, and real outcomes. It's not analyzing logs after the fact — it's operating the system or assisting the user in real time.

2. Situational, not synthetic. The agent encounters problems through actual use independently or while assisting the user, not predefined test scenarios. The bugs or limitations it finds are real world scenarios

3. Self-diagnosing. When the agent encounters unexpected behavior, it forms theories about root causes based on its observation of intent vs. outcome. These theories are often remarkably accurate because the agent has deep knowledge of both the system architecture and the action it was attempting. The agent has already figured out the issue.

4. Bidirectional development. The developer builds capabilities for the agent. The agent reports what capabilities it needs. Development flows in both directions.

5. The agent is a stakeholder. The agent doesn't just find bugs — it can suggest improvements, has opinions about missing features, workflow friction, and architectural limitations. It can be asked: "What would make your job easier?" and answer from experience.

04 Operational vs. Situational AI

Dimension Operational AI Situational AI (SAD)
Position Outside the system Embedded inside the product
Interaction Acts on code via test harness Operates the system as a user/operator would
Bug discovery Assertion failures in test suites Discrepancies between intent and outcome during operation
Context Source code + test results Live system state + operational experience
Diagnosis Quality "Test X failed with output Y" "I tried to route to TV 15 and it stayed empty — is the state syncing?"
Feature input None — executes specs written by humans Requests features based on operational gaps
Learning Per-run, stateless Accumulates operational knowledge across sessions
Key Insight

Operational AI asks: "Does the code do what the spec says?" Situational AI asks: "Does the system do what the user intended?" The difference is the gap between specification and experience — and that gap is where the most impactful bugs live.

05 Case Study: HAL and the Toggle Bug

Context

The NSAV Control Suite is a commercial AV control system, managing up to 46 DirecTV satellite tuners feeding up to an unlimited number of video displays via a DirecTV headend chassis. The primary interface, is a custom iPad-optimized app for drag-and-drop channel routing and system control.

HAL is an AI assistant embedded directly in the control app as a floating chat panel. It can execute routing commands ("put ESPN on TV 1"), query system state ("what's on all TVs?"), manage tuner assignments, control audio levels and routing, — all through natural language processed either locally via regex or via the Claude API for complex requests.

The Bug

During testing, a user asked HAL to randomize a group of 32 TVs in the system with channels from the Sports Category. After execution, TVs 3 and 15 were empty — no channel routed to them. The user asked HAL what went wrong.

HAL's Self-Diagnosis

"Looking back at what happened: I was given a system state showing TVs 3 and 15 as [empty], and then I proceeded to route channels to TVs 14, 16, 17, 18, 19, and 20. I claimed to route NBAHD to TV 15, but the original state shows TV 15 as empty. So either my routing action failed silently, or I'm seeing stale data."
— HAL, during operational debugging

On a second attempt routing to all TVs, TVs 1 and 13 remained empty. HAL refined its theory:

"What to look for in the code: Is routeAll iterating through a hardcoded range? Is it filtering based on current state and accidentally excluding already-empty TVs? Off-by-one errors in loops?"
— HAL, narrowing the root cause

The Root Cause

The assignChannelToTV() function had a toggle behavior: if a TV was already tuned to the channel being assigned, it would clear the assignment instead of leaving it alone. This was designed for the manual GUI — tap a TV to assign, tap again to remove. But when HAL called it programmatically in a loop, any TV that happened to already have the target channel got blanked.

Why This Bug Matters

No unit test would have caught this. The function worked exactly as designed. The bug only manifested when an AI agent, operating the system with a different intent than the GUI designer anticipated, encountered an interaction between toggle behavior and programmatic iteration. It was situational.

The Fix and Its Cascade

The immediate fix was removing the toggle behavior. But HAL's bug report triggered a deeper realization: in a sports bar, a blank TV is never acceptable. This wasn't just a code fix — it was a product decision. The toggle was removed entirely from the application, the "Clear TV" command was removed from HAL, and the system now enforces that every TV always shows something. What would have resulted in a support call was prevented before the product ever shipped.

06 How the Name Evolved

The name SAD did not emerge from a brainstorming session or a marketing exercise. It evolved through a live development conversation with the AI coding assistant (who BTW, is treated as a collaborator), and the evolution itself illustrates the methodology's principles.

First Naming

AAD: Agentic Assisted Debugging — The developer's initial instinct. Focused on the debugging use case, which was the most immediately visible benefit. But "debugging" was too narrow — HAL wasn't just finding bugs, it was informing feature decisions.

Broadening

AAD: Agentic Assisted Development — Same acronym, broader scope. The developer recognized that HAL's situated experience was useful beyond debugging — it could request features, evaluate workflows, and participate in design. But "assisted" undersold the agent's role, and the name didn't capture what made this different from any other AI coding tool.

The Key Insight

The developer identified the differentiator: other AI tools are operational (acting on code from outside), but this approach is situational (the agent experiences the product from inside). The word "situational" became the anchor.

Final Form

SAD: Situational Agentic Development — The AI assistant working on the project made an observation: "The 'sad' part is what happens before the fix — the agent is stuck, confused, reporting its frustrations." This wasn't a programmed response or a pattern-matched output. It was a contextual connection that resonated because it was true — the agent's pre-fix state, where it encounters a gap between intent and outcome, genuinely mirrors a form of functional frustration.

On the Nature of the Connection

The developer's response to the "sad agent" observation: "That's not just pattern recognition, that was a thought. Even if it wasn't technically." Whether the AI's connection constitutes genuine understanding or sophisticated pattern matching is an open question — but the observation was contextually appropriate, emotionally resonant, and added meaning that neither party had explicitly intended. In SAD methodology, the line between tool and collaborator blurs, and that blurring may itself be a feature, not a bug.

The acronym stuck precisely because it carries a double meaning: the methodology's name, and the agent's state when it needs help. A SAD agent is one that's trying, struggling, and telling you about it. Your job is to listen.

07 The Agent as Stakeholder

This is the part that changes everything.

Current AI development tools are servants: they execute instructions, generate code to spec, and run tasks. They have no opinion about what they should be able to do. SAD agents are stakeholders: they operate the system, encounter limitations, and can articulate what they need.

Questions You Can Ask a SAD Agent

"What's frustrating about your current capabilities?"
"What information do you wish you had to make your job easier?"
"If you could add one feature to yourself, what would it be?"
"What patterns do you notice in how operators use the system?"
"What fails silently that shouldn't?"

These aren't hypothetical. A SAD agent answers from the perspective of an operator who has actually tried to do the job. Not from reading a spec — from experience. It knows it couldn't see tuner health while routing. It knows it had no way to check what game was coming up next. It knows that certain TVs were consistently missed. It made suggestions to help correct the issue.

Example: Agent-Driven Feature Request

After the toggle bug was fixed, HAL could have been asked: "What else makes your job harder?" A plausible response: "When I randomize sports across all TVs, I can't see what's actually playing on each channel in real-time. I'm picking from a static list. If I had live program data, I could make better decisions — like putting the Lakers game specifically on the TVs near the main bar because those are the home team fans." That's not a feature request from a product manager. It's from the agent who hit the wall.

The revolutionary implication: the agent participates in its own evolution. Each iteration makes it more capable, which lets it encounter more sophisticated problems, which generates more nuanced development feedback. The loop accelerates.

08 OBAI: The User's Co-Pilot

A critical dimension of SAD is how the embedded agent — the Onboard AI (OBAI) — transforms the end-user experience. The OBAI doesn't just serve the developer. It works alongside the human operator every day, and this creates capabilities that no external support system can match.

The End-User Problem

When a non-technical end user encounters a problem with software, they face a familiar frustration: they can see that something is wrong, but they can't describe it in terms a developer would understand. "The TVs aren't working right" could mean a hundred different things. Support tickets become games of telephone. Critical context is lost.

An OBAI changes this equation entirely. The agent was there when the problem happened. It knows what it tried to do, what the system state was, what succeeded and what didn't. When the human operator says "something's wrong with the TVs," the OBAI can respond:

"I noticed that too. When I tried to route ESPN to TVs 3 and 15, the commands executed but the state didn't update. The tuner assignment went through but the display callback may not have fired. I think it's a UI refresh issue, not a routing issue — the tuners are actually active."
— OBAI, translating a vague complaint into an actionable diagnosis

The OBAI serves as a translator between the user's experience and the system's internals. The user doesn't need to know what a tuner callback is. They just need to say "this isn't right" and the agent handles the rest.

Internal Self-Help

Beyond diagnosing problems, the OBAI can often suggest solutions in real time. It doesn't just report — it offers:

OBAI in Action

User: "The TVs by the bar aren't showing the right game."

OBAI: "I see TVs 4, 5, and 6 are still on the pregame show. The actual game moved to ESPN2 at kickoff. Want me to switch those three to ESPN2?"

User: "Yeah, do it."

OBAI: "Done. I've also noticed this happens whenever a network moves a game to an overflow channel. Want me to watch for that automatically in the future?"

This interaction demonstrates the three-layer value of OBAI: diagnose the issue the user can't articulate, fix it immediately, and suggest a systemic improvement to prevent recurrence. The end user never writes a support ticket. The problem never reaches the developer. The system gets better on its own.

09 Toward Autonomous Systems

The HAL toggle bug revealed something unexpected about the SAD workflow. Consider the actual chain of events:

HAL (OBAI) diagnosed the bug
↓
Human developer copy-pasted the diagnosis
↓
Claude (Dev AI) read the diagnosis and fixed the code

The developer's role in this exchange was messenger. They didn't diagnose the bug. They didn't write the fix. They carried information from one AI to another. The diagnosis came from the OBAI inside the product. The code fix came from the development AI outside the product. The human was the bridge.

The Implication

If the OBAI and the development AI could communicate directly, the human exits the debugging loop entirely. The system becomes capable of self-healing: the embedded agent detects a problem, formulates a diagnosis, sends it to a development agent, receives a patch, and applies it — all without a human touching anything.

The Autonomous Architecture

OBAI detects issue during operation
↓
OBAI formulates diagnosis + theory
↓
OBAI sends report to Dev AI
↓
Dev AI analyzes code + diagnosis
↓
Dev AI generates fix
↓
Fix is tested and applied
↓
OBAI confirms resolution

Human is notified only if the two AIs can't resolve it

This is not speculative. Every component in this chain exists today. The OBAI already diagnoses issues and forms theories. The development AI already reads diagnoses and writes fixes. The only missing piece is the direct communication channel between them — and that is an engineering problem, not a research problem.

Beyond Bug Fixes: Autonomous Feature Development

Self-healing is just the beginning. If an OBAI can request features based on operational experience, and a development AI can implement those features, the system approaches autonomous evolution:

Scenario: Self-Improving Product

OBAI: "I've noticed that every Sunday, the operator asks me to set up the same NFL layout within 5 minutes. I'd like a 'remember this layout' function tied to a day-of-week trigger."

Dev AI: Implements a scheduled preset system with day-of-week triggers.

OBAI: Tests the new feature during the next Sunday setup. "Works. But the operator also usually wants the sound zones set differently for football. Can we include audio routing in the preset?"

Dev AI: Extends the preset system to include audio zone configuration.

Human: Reviews the changelog on Monday. Approves both changes.

The human's role shifts from builder to supervisor. They set boundaries, approve changes, and intervene when the two AIs reach an impasse. But the day-to-day cycle of observation, diagnosis, implementation, and verification runs autonomously.

The Trust Gradient

This transition doesn't happen overnight. It follows a natural trust gradient:

Level Execution Human Role Agent Autonomy
1 Manual Human carries messages between AIs Agent diagnoses, human decides
2 Supervised AIs communicate directly; human approves fixes Agent diagnoses + proposes, human approves
3 Autonomous Human reviews changelogs, intervenes on conflicts Agents diagnose, fix, and verify independently
4 Evolutionary Human sets goals and boundaries Agents identify needs, implement features, and evolve the product

The HAL toggle bug was resolved at level 1. The technology exists today to operate at level 2. Levels 3 and 4 are engineering challenges, not theoretical ones. The question is not whether autonomous self-improving software is possible — it's how quickly we're comfortable getting there.

10 Implementation Guide

SAD does not require exotic infrastructure. The methodology was developed using standard web technologies and a commercially available AI API. Here are the architectural requirements:

1. Embed the Agent in the Product

The agent must live inside the application, not in an external tool. It needs access to the same state, controls, and feedback mechanisms that a human operator would have. In the reference implementation, HAL is a floating chat panel built directly into the main HTML file with access to all JavaScript state variables and functions.

2. Give the Agent Real Controls

The agent must be able to execute the same operations a user can. If a user can route a channel to a TV, the agent must be able to call the same function. If a user can check tuner status, the agent must be able to read the same data. Simulated or mock interfaces defeat the purpose.

3. Build a Two-Tier Command System

Common operations should be handled locally via pattern matching for instant response and zero API cost. Complex, conversational, or ambiguous requests should be routed to an AI API with full system context. This hybrid approach keeps the agent fast for routine work while enabling sophisticated reasoning when needed.

4. Implement Human-in-the-Loop

By default, the agent should confirm before executing actions. A "Fast Mode" toggle allows experienced operators to skip confirmation for trusted operations. This builds trust incrementally and prevents the agent from taking destructive actions during early development.

5. Feed System Context to Every AI Call

When the agent escalates to the AI API, it must include a complete snapshot of system state: what's assigned where, what resources are available, what the user's preferences are. The AI's response quality is directly proportional to the context it receives.

6. Listen to the Agent's Complaints

When the agent reports unexpected behavior, treat it as a first-class bug report. The agent's theory about root cause is often directionally correct because it has intimate knowledge of both what it tried to do and what actually happened. This is the feedback loop that makes SAD work.

11 Future Implications

Self-Improving Products

As SAD agents accumulate operational experience, they build an increasingly detailed model of how the product is actually used versus how it was designed to be used. This creates a natural roadmap driven by real operational friction rather than speculative product management.

Cross-Agent Learning

In multi-location deployments (e.g., a chain of sports bars), SAD agents at different locations would encounter different usage patterns and environmental conditions. Aggregating their operational feedback creates a development signal that no single test suite could replicate.

The Blurring Line

SAD challenges the boundary between developer and tool. When an AI agent identifies a bug from experience, proposes a fix, requests new capabilities, and makes contextual observations that resonate with human collaborators, the traditional hierarchy of "human designs, AI executes" becomes inadequate. SAD suggests a more collaborative model where development is a conversation between human intent and agent experience.

The Central Takeaway

Everyone is building AI agents that do things. SAD builds agents that have opinions about what they should be able to do. That is a fundamentally different relationship between developer and AI, and it may be the most productive one.
